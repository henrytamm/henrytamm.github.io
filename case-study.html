Automating documentation workflows with AI

How I built an AI-powered system that reduced routine documentation tasks from 2-3 hours to under 10 minutes — and got it adopted by the entire writing team.

The problem

Technical writers on the Data 360 team spent significant time on repetitive, structured tasks — drafting release notes from Jira tickets, formatting DITA XML boilerplate, looking up metadata for new topics, and routing doc requests from Slack.

These tasks weren't intellectually hard, but they were time-consuming and error-prone. A single release note could take 2-3 hours when you factor in context switching: reading the ticket, finding the right template, formatting XML, checking edition tags, looking up related topics, and submitting for review.

Before
  • 2-3 hours per release note
  • Manual template lookup each time
  • Copy-paste metadata from old topics
  • Slack requests lost in threads
  • Inconsistent formatting across writers
After
  • 5-10 minutes per release note
  • Auto-generated from ticket context
  • Metadata populated automatically
  • Structured intake via slash command
  • Consistent output every time

My approach

I didn't start by building tools. I started by documenting my own workflow for a week, noting every repeated action, every copy-paste, every "where is that template again?" moment. Then I categorized them:

  • Automatable: Release note drafting, XML formatting, metadata insertion, tag flipping
  • Semi-automatable: Doc intake routing, complexity classification, writer assignment
  • Needs a human: Technical accuracy review, audience judgment, information architecture

The key insight: AI is great at structured generation (turning inputs into formatted outputs) but bad at editorial judgment. So I built tools that handle the former and leave the latter to writers.

What I built

A suite of AI-powered skills integrated into the team's existing workflow:

1. Doc Request Intake — A slash command that accepts natural language requests, classifies complexity (Small/Medium/Large), asks only the missing questions (max 3), then creates a work item and posts to the intake channel. Zero friction for stakeholders.

2. Release Note Generator — Takes a work item or ticket and generates a complete DITA XML release note: title, shortdesc, edition tags, section structure, and metadata. Follows the team's style guide rules automatically.

3. Tag Management — Automates beta/pilot/GA tag flipping across ditamaps when features change status. What used to require manually finding and editing 5-10 files now happens in one command.

Design principle: Every tool I built produces output that a writer can edit — not output that replaces the writer. The AI handles the 80% that's mechanical; the writer handles the 20% that requires judgment.

Technical decisions

  • AI skills over standalone scripts: Built as slash commands inside the team's existing AI coding tool, not separate apps. Zero new tools to learn.
  • No confirmation gates: The intake skill submits immediately after gathering info. Every "does this look right?" prompt adds friction that discourages people from filing requests.
  • Messaging-first architecture: The Slack post is the critical path, not the work item. A visible post without a tracker ticket still gets picked up. The reverse is an orphaned ticket nobody sees.
  • Graceful degradation: If one integration fails (auth expired, API down), the tool still completes what it can and tells you what it couldn't do. Never blocks.

Results

  • Routine documentation tasks dropped from 2-3 hours to 5-10 minutes
  • Adopted by the full writing team — not just the person who built it
  • Stakeholders actually file doc requests now (friction went to near-zero)
  • Formatting consistency improved — no more "which template do I use?" guesswork
  • Freed up writer time for higher-value work: information architecture, audience analysis, technical accuracy

What I learned

The hardest part wasn't the technical implementation — it was understanding what should and shouldn't be automated. Early versions tried to do too much (auto-publishing, auto-routing to specific writers) and the team didn't trust the output.

The version that stuck was the one that treated AI as a drafting assistant, not a replacement. Writers stay in control. The AI just removes the tedious parts so they can focus on what they're actually good at.

Built with HTML, CSS, and vibes. No frameworks were harmed.
